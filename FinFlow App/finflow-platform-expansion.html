FinFlow · Architecture Reuse Map

One platform.
Many problems.

Strip away the financial data and the score engine. What remains is a shared household dashboard, a daily habit loop, peer accountability circles, and a zero-CAC institutional channel.

That four-component architecture solves an entirely different set of problems in every industry where households make decisions together under stress.

The Core Architecture · Unchanged Across All Verticals
01
Shared Household Dashboard
Both decision-makers see one live view. Real-time sync. Partner invite. Solo mode with delayed join. Privacy controls per item.
02
Daily Habit Loop
Morning signal → midday nudge → evening check-in → weekly review. One notification per day maximum. Anchored to existing moments.
03
Accountability Circle
8-unit peer groups matched by life stage. Monthly facilitated sessions. Social commitment mechanism. Churn drops 80% on join.
04
Institutional Channel
Employer or school distributes the product at zero CAC. One contract = 200+ users activated in 30 days. The moat no competitor can copy.

The Reuse Thesis

Replatforming is a frontend decision,
not a rebuild.

✓ Stays Identical · Zero Re-Engineering
Auth + Household Creation
User registration, partner invite, household account model. Copy-paste to any vertical.
Real-Time Partner Sync
WebSocket infrastructure that updates both screens in under 3 seconds. Domain-agnostic.
Notification Engine
One-per-day rule. AI selects highest-relevance signal. Morning / midday / evening slots. Swap the content, not the system.
Circle Matching Algorithm
8-unit peer group formation by life stage + geography + shared context. Parameter changes, engine stays.
Weekly Review Flow
10-minute guided structured session. Wins → worry → one action. Template variables swap per domain.
Employer / School Portal
Admin dashboard, aggregate engagement metrics, billing. Relabel "employer" per channel partner.
Goal Setting + Progress Bars
Target → milestone → celebration mechanic. Change the target type, keep the dopamine loop.
Onboarding Flow (7 Questions)
Conversational intake that routes to right starter kit. Rewrite questions, keep the routing logic.
↻ Changes Per Vertical · Frontend + Data Layer Only
Dashboard Data Points
Financial transactions → health logs / learning records / care tasks. Data schema changes. UI layout identical.
Third-Party API Integrations
TrueLayer → NHS API / school MIS / wearable SDK / HR system. One integration swap per vertical.
Notification Copy + Triggers
"You spent £340 on subscriptions" → "Your child missed 3 homework deadlines." Content layer only.
Circle Matching Parameters
Income tier + life stage → diagnosis type + care load + school year group. Parameters change, algorithm stays.
Channel Partner Type
Employer / school → NHS surgery / housing association / law firm / hospice. Same portal, different partner logo.
Visual Language + Branding
Colour palette, iconography, terminology. Design tokens swap. Component library stays.
Regulatory Compliance Layer
FCA → CQC / Ofsted / ICO / SRA depending on domain. Compliance wrapper changes. Data model mostly reusable.
Starter Kit Content
90-day financial plan → 90-day care plan / school readiness plan. Content swap, not structure.

The architecture is the moat.

FinFlow's engineering investment doesn't depreciate across verticals — it compounds. Every new deployment on the same infrastructure lowers the marginal cost of the next one. The household OS gets cheaper to build and harder to replicate with every domain added.
