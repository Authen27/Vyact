# FinFlow — Database Migrations

> Source of truth for the Postgres schema is the ordered set of migration files
> under [`supabase/migrations/`](../supabase/migrations/). The single-file view
> at [`db/schema.sql`](schema.sql) is a **generated** human-readable concat of
> every migration and is regenerated by tooling — do not edit it by hand.

Owner: Engineering. Adopted: 2026-05-23 (Remediation PR #3 / **TD-18**).
Governance: [`docs/TEST_GOVERNANCE.md`](../docs/TEST_GOVERNANCE.md).

---

## 1. Why this exists

Before PR #3, the schema lived as one hand-edited file (`db/schema.sql`) that
operators ran in the Supabase SQL editor. That meant:

- No version history of schema changes (every commit looked like "edit one big
  file").
- No way to know which environments had which changes applied.
- Privileged surface (admin RPCs / `admin_roles`) was already referenced by
  the admin app but **not** in the committed schema — see TD-04. Untracked
  changes is exactly the failure mode this directory eliminates.

Adopting a migrations directory makes every schema change a discrete,
reviewable, applied-in-order unit, with a single CI gate (`db-migrations`)
that refuses drift.

## 2. Layout

```
supabase/
  migrations/
    00000000000000_initial_schema.sql      # baseline — current production shape
    <YYYYMMDDHHMMSS>_<snake_name>.sql      # every subsequent change
db/
  schema.sql                                # GENERATED — concat of the above
  MIGRATIONS.md                             # this file
```

**Naming rules** (enforced by `scripts/db-migrations-check.mjs`):

| Rule | Why |
|---|---|
| Filename matches `<14+ digit timestamp>_<snake_case_name>.sql` | Lexicographic order = chronological order. Supabase CLI convention. |
| Timestamps are unique across all files in the directory | Prevents ambiguous ordering. |
| Each file is non-empty | Catches accidental commits. |
| `db/schema.sql` exactly equals (header) + concat of all migrations | Prevents the snapshot from drifting away from reality. |

`00000000000000` (14 zeros) is a deliberate, conventional sentinel for the
**baseline** migration — it sorts before any real timestamp, so the order
is always: baseline first, then everything else by date.

## 3. Adding a new migration

```bash
# 1. Pick a UTC timestamp and a short snake_case name.
ts=$(date -u +%Y%m%d%H%M%S)
name=add_admin_roles_table

# 2. Create the migration file.
cat > "supabase/migrations/${ts}_${name}.sql" <<'SQL'
-- TD-04 — version-control the admin privilege surface.
create table if not exists admin_roles ( … );
-- … RLS policies … etc.
SQL

# 3. Regenerate db/schema.sql.
node scripts/db-migrations-check.mjs --fix

# 4. Validate (this is what CI runs).
node scripts/db-migrations-check.mjs

# 5. Apply to staging (uses the Supabase MCP `apply_migration` tool per
#    the PR template). Confirm `get_advisors` shows no new warnings.
```

**Migrations must be additive.** No `drop table`, no destructive
`alter column ... type`, no data-truncating updates without a paired backup
plan. Reversible by appending another migration, never by editing history.

---

## 4. Applying migrations

| Environment | How | When |
|---|---|---|
| **Staging Supabase project** | `apply_migration` MCP tool, supplying the migration filename + body | Whenever a migration PR is opened, before requesting review |
| **Production Supabase project** | Same `apply_migration` tool, by the release owner | After the PR merges to `main` and the Deploy workflow has shipped the app code that depends on the schema |
| **Local dev (cloud-mode)** | Same tool, against your local/personal Supabase project | When you first pull a PR that changes the schema |

The previous "open the SQL editor and paste `db/schema.sql`" path is
deprecated for new operators and only safe for fresh, never-applied
environments. Once any migration has been applied, only `apply_migration`
should be used.

---

## 5. The CI gate

`scripts/automation-run.mjs` invokes `node scripts/db-migrations-check.mjs`
as the **db-migrations** gate. It fails on:

- A filename that doesn't match the naming rule.
- A duplicate timestamp.
- An empty migration file.
- `db/schema.sql` differing from the regenerated concat (line-ending
  normalised, so Windows CRLF checkouts pass).

If the gate fails with "schema.sql has drifted", the fix is mechanical:

```bash
node scripts/db-migrations-check.mjs --fix
git add db/schema.sql
```

---

## 6. Out of scope today (tracked elsewhere)

- **Applying** migrations from CI itself (requires a Postgres service on the
  runner). Not blocking — the MCP `apply_migration` flow against staging is
  the current discipline, and the static gate covers the most common drift
  classes.
- **Down migrations / rollbacks.** Additive-only is the rule; rollback is
  by additive forward fix, not by reverting history.
- **Type-generation** from migrations into the TypeScript adapter. Future
  ratchet; not blocking for PR #3 or PR #4.
