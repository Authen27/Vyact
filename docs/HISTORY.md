> **Consumer v9.3.3 status — Budget identity owned by the database (2026-06-14).** The definitive fix, and it **retires the v9.3.1 deterministic-id approach** (`lib/budgetIdentity.ts` + CON-UNIT-065 were deleted). That approach coupled the budget PK to its identity and broke twice: delete+recreate a month landed on the soft-deleted same-id row via `ON CONFLICT (id)` and never cleared `deleted_at` (budget came back invisible); and recovered random-id rows collided with fresh deterministic ids on `uq_budget_month`. Goals/debts/assets were never affected because they mint a random id per create and have no identity index — budgets were fragile *because of* the client hack. **Identity now lives in the DB** behind one writer: `upsert_budget(h, b, mode)` RPC (`20260614140000_v933_upsert_budget_identity_authority.sql`). `create` = `INSERT … ON CONFLICT (identity) WHERE deleted_at IS NULL DO NOTHING` → if nothing inserts, raise `BUDGET_EXISTS` (errcode 23505 — race-proof, fires for another member's unsynced budget too); `replace` = `DO UPDATE …, deleted_at=NULL` (idempotent/revive). The DB mints the id. **Client:** `DataAdapter.createBudgetChecked` (Supabase→RPC, Local→existence check, Hybrid→online-first then cache); `store.upsertBudget` create-branch calls it; `BudgetFormModal` catches `BudgetExistsError` → "already exists, refreshing" + `manualRefresh`; edits keep the TD-03 update-by-id path. **Testing:** validated by the zero-cost **auto-rollback `DO`-block scenario harness** (create/duplicate-reject/delete+recreate/replace-converge/replace-revive/annual, all PASS).

> **Consumer v9.3.2 status — Budget create reaches the cloud (2026-06-14).** Follow-up to v9.3.1: a NEWLY created budget still didn't sync — the create never reached the cloud. **Root cause:** `budgets.period` is a legacy `NOT NULL DEFAULT 'monthly'` column; the v9.1 scope-based form never sends `period`, but `budgetToRow` mapped the missing value to an **explicit `null`** → every new-budget INSERT violated NOT NULL → the optimistic write threw, retried, and **dead-lettered**. **Fix:** `budgetToRow` defaults `period` to `'monthly'`. **Convention (still binding):** when mapping a domain object to a row, a NOT-NULL column with a DB default must be written as its default or **omitted** (so the default applies) — never as an explicit `null` (`provToRow` does this right via `?? undefined`, which supabase-js drops). +CON-UNIT-066.

> **Consumer v9.3.1 status — Budget multi-device convergence, root-cause fix (2026-06-14).** Budgets weren't syncing across a household's devices. **Root cause:** a budget's identity is `(household, scope, year, month)` — enforced by `uq_budget_month`/`uq_budget_annual` — but the client minted a **random `uid()`** per new budget, so two devices creating the same period produced two PKs for one slot; the second `INSERT` violated the unique index, retried, and **dead-lettered** (silent cross-device loss). First attempt (deterministic container id) was superseded by v9.3.3's DB-owned identity. **DB** (`20260614130000_v931_budget_identity_convergence.sql`): `replace_budgets` rewritten (it had stripped `scope/period_year/period_month` on restore and PK-collided on same-id re-insert) → schema-correct + `ON CONFLICT (id) DO UPDATE`; dropped the legacy competing `budgets_household_category_uniq` index.

> **Consumer v8.1.0 status — Ask Vyact assistant.** A deterministic, on-device, **no-LLM** three-bucket assistant (Capture / Interpret / Forecast). Five-stage pipeline: `normalise → entityExtract → classifyIntent → resolve → phraseResponse`. The assistant phrases; services compute (stage 4 `resolve` is the only place money is computed, via the same dashboard services). Two seams (`classifyIntent`, `phraseResponse`) make a future LLM a drop-in; stages 1/2/4 are pure and model-agnostic. Files in `lib/askVyact*.ts`, wired in `pages/Chat.tsx`.

> **Consumer v8.0.0 status — Onboarding & Activation module.** Per-household onboarding: six-step segment-driven flow capturing a minimal baseline whose estimates converge to confirmed over a 21-day window. The whole module is a clean no-op when `isOnboardingEnabled()` is false. Onboarding is owned by the HOUSEHOLD and cloud-synced: state machine on `households.onboarding` (jsonb) + a localStorage cache; record provenance as `confidence`/`source`/`estimated_at`/`confirmed_at` columns on entity tables. Honest data: any value with `confidence !== 'confirmed'` renders `<EstimatedTag/>`. Migration `20260606120000_v8_onboarding_state.sql`.

> **Consumer v6.4.1 status.** All 10 pages ported to React.
