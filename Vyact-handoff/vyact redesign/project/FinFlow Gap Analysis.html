CriticalCU‑01
Onboarding · step 4 → Dashboard

Onboarding ends before the value moment

The flow drops the user on a Dashboard populated with zeros. The PRD's NorthStar KPI timeToFirstTxnSec < 90s can't be hit if step 5 (add a transaction) doesn't exist.

Impact

Empty dashboards cause D1 churn. Admin currently shows timeToFirstTxnSec 70s — likely measuring sessions where users did figure it out, ignoring those who bounced.

Recommendation

Add step 5: "Add your first transaction" with three big chips (Salary · Rent · Groceries) — pre‑filled amounts editable inline. Step 6: "Invite your partner" (skippable). Promote the existing skip‑link to a secondary action so it isn't the path of least resistance.

WatchCU‑02
Onboarding · concern selection

Concern selection has no observable effect on the dashboard

The copy promises "This reorders your dashboard for what matters most to you" but Dashboard.tsx renders the same fixed grid for everyone — Pulse, 4 cards, budgets, transactions, goals, donut, net worth, debt.

Impact

Trust erosion. Users notice the broken promise within 30 seconds. Worse: it makes every other claim ("Pulse Score weighted for your situation") feel suspect.

Recommendation

Hoist the panel matching profile.primaryConcern to position 2 (after Pulse). Spending → Budget Progress; Debt → Debt Overview; Savings → Goals; Retirement → Net Worth. Or remove the promise.

PolishCU‑03
Onboarding · step 1

Skip default goes to "Family" — wrong for ~60% of signups

Mock data shows tier mix tilts heavily Free + Family, but PostHog templates list 6 segments evenly: young_couple, family, single, self_employed, retiree, student.

Impact

Skippers get wrong starter budgets/categories — silent friction during week 1.

Recommendation

Detect from sign-up email domain + browser hints; fall back to "Single" (safest blank slate). Or remove skip entirely — 4 steps × 90s is not too much friction for the lifetime relationship.

1.2 — Daily operations: data entry & categorisation
CriticalCU‑04
Dashboard · header · Transactions · header

"Add Transaction" buttons are decorative

Both pages render <Button><Plus/> Add Transaction</Button> with no onClick handler. There's no add-transaction modal anywhere in the React app.

Impact

The single most-frequent action in the app is non-functional in the modern stack. Users either give up or fall back to the v5 vanilla shell — an eject the product can't afford.

Recommendation

Ship AddTransactionModal with: type segmented control · amount · category picker (suggest from history) · payment method · split toggle · private toggle · member · date · note. This is the P0 React migration — every other page block is downstream of it.

CriticalCU‑05
No bank / aggregator import

Manual entry is the only path in

CLAUDE.md acknowledges "No bank aggregation (open banking / Plaid) — Phase 2." But every consumer-finance app the user's compared against (Mint, Monarch, Lunch Money, YNAB) imports transactions on day one.

Impact

D7 retention will cap around 30–35% no matter how good the UI is, because the chore of manual entry compounds daily. d7RetentionPct > 60% target is unreachable without it.

Recommendation

Even if Plaid is Phase 2, ship CSV import + email-receipt forwarding + Apple Pay / Google Pay receipt photo OCR in v7.5. Position manual entry as the rare case ("missed by import").

WatchCU‑06
Transactions · row

No bulk operations · no inline edit · no recategorise

TxnRow is read-only when showActions is false; even with actions, there's no multi-select, no "recategorise these 5", no "mark all groceries Tesco as 'Groceries'" rule.

Impact

Once aggregator imports land, users will have 50–500 uncategorised rows on day 1. Without bulk edit they bounce.

Recommendation

Add row checkboxes + a sticky bulk-action bar (Categorise · Tag · Mark private · Delete). Add "Always categorise [merchant] as [X]" rule capture from any single edit.

WatchCU‑07
Transactions · search

Search is substring-only · no amount/date range

Filters cover type, category, month, member. No amount filter (>$200), no date range, no payment-method filter, no "private only".

Impact

"How much did I spend at Whole Foods last quarter?" requires Reports + math. Friction → users stop exploring → less stickiness.

Recommendation

Add a saved-views row above filters (Last 30d · This year · Over $500 · Private). Saved searches power the daily check-in habit.

1.3 — Planning & insight
WatchCU‑08
Planner · all recommendations

Planner has no "do it" path

Each recommendation has a single deep link (r.action.route) to a related page. Nothing actually executes the recommendation. A "pay $200 extra to highest-APR debt" rec doesn't know how to schedule that payment.

Impact

recsFollowedPct targets 25%. Without one-click execution it'll trend <10%. Planner becomes "yet another things-to-feel-bad-about list".

Recommendation

Each rec gets a primary action button: "Schedule extra payment", "Create budget", "Set up recurring transfer". Wire to existing Recurring + Debts forms. Track click-through to closed-loop "did the user do it within 7 days".

WatchCU‑09
Chat · stub backend

Chat replies from a local pattern matcher and admits it

The privacy notice says "currently in stub mode — answers come from a local pattern-matcher." Users see this and stop using Chat permanently within 2 sessions. avgChatSatisfaction > 4.2 is unmeasurable on a stub.

Impact

Sets a low ceiling on first impression. v8.1 wiring of the real LLM may not recover users who already learned "Chat doesn't work".

Recommendation

Don't ship Chat publicly until the Edge Function is live. If you must, gate behind a "Beta · early access" opt-in toggle and don't show a placeholder reply for queries the matcher can't handle — say "I can't answer that yet, this is queued for our LLM upgrade in [date]."

PolishCU‑10
Reports · period selector

5 period buckets · no custom range · no compare

Users want "this Nov vs last Nov", "trip to Italy vs baseline", "before vs after raise". The 5 fixed buckets (D/W/M/Q/Y) don't enable comparison.

Impact

Reports become a destination only at month-end, not a habit.

Recommendation

Add date-range picker + "vs previous period" + "vs same period last year" toggle. Persist as saved views.

1.4 — Multi-user & households
CriticalCU‑11
Households · invite flow

Invitations require manual link-sharing

InviteModal creates the row, then says "Configure the send-invite-email Edge Function in Supabase to email this automatically." Today the inviter has to copy the link and DM/text it.

Impact

multiMemberPct > 40% is the single best retention lever (multi-user households churn ~3× less). Manual link sharing roughly halves invite-acceptance.

Recommendation

Ship the Edge Function before launch. Add SMS option as backup. Add "Tap to share" sheet on mobile that prefers iMessage/WhatsApp where SMS isn't installed.

WatchCU‑12
Households · activity log

Activity log is admin-grade · not human-grade

Entries render as <Action: txn.create> on transaction. A partner glancing at the household feed sees postgres event names, not "Priya added $42 at Whole Foods (Groceries) yesterday."

Impact

The "kitchen-table conversation" promise dies in this list. Households are about seeing each other's money moves, not auditing them.

Recommendation

Re-template every action to natural language. Group by day. Show actor avatar + amount + category icon. Differentiate "Priya recorded" from "Priya edited yours" from "System auto-confirmed rent" — different visual weights.

WatchCU‑13
Households · child role

'Child' role exists in code, has no UI

Permissions table grants children scoped permissions. There's no kid-friendly view, no allowance flow, no "ask parent to approve" pattern. Inviting a child as 'child' currently delivers them the adult dashboard with most buttons disabled.

Impact

Family segment loses its biggest differentiator (financial education for kids). Competitors (GoHenry, Greenlight) own this.

Recommendation

v7.6 minimum: child sees a stripped-down "My Money" page (allowance balance, this week's spending, savings goal jar, recent transactions). Parents see a "Kids" admin tab in Households.

PolishCU‑14
Households · transfer ownership

RPC exists; no UI surfaces it

Schema has a transfer_ownership RPC and the 'transfer_ownership' permission, but no button anywhere lets an owner do this. Sole-owner-leaves flow has a TODO.

Impact

Edge case — but breakups, family restructuring, and admin handoffs are real. Today the owner has to delete + reinvite (data-loss risk).

Recommendation

Add "Transfer ownership" to Danger Zone, gated to current owner, with confirmation including 2FA-style re-auth.

1.5 — Stubbed pages still pointing to v5 vanilla
Migration debt

Seven pages — Budgets, Goals, Splits, Debts, Net Worth, Settings, Help — render a placeholder with a v5 deep link. Three of these (Budgets, Goals, Net Worth) are surfaced as Dashboard panels that link back to themselves as stubs. The Pulse Score weights Debt Health 20% and Goal Progress 15% — both calculated from data that lives behind stubbed pages.

CriticalCU‑15
Stubs · /budgets · /goals · /debts · /networth

Dashboard cards link to non-functional pages

Click "View all" on Dashboard's Budget Progress → land on a "React migration in progress" page with a v5 link. Same for Goals, Net Worth, Debts. Pulse Score advertises 20% from Debt Health, but the user can't see or edit debts in the same app.

Impact

Dashboard becomes a teaser. Pulse becomes opaque ("why is mine 67? where do I fix it?"). The two-app handoff feels like a bug.

Recommendation

Either: (a) finish the React migration before public launch — budgets and net worth share data shapes already in store.ts, this is hours not days; or (b) hide the stub-linked panels for now and don't promote Pulse Score as a key metric until its inputs are editable.

WatchCU‑16
Stubs · /settings

Settings page is a stub — but theme toggle is in the sidebar

Theme switcher and "Sign out" live in the sidebar footer, not in /settings. There's no React-side way to edit profile, change base currency, edit FX rates, change date format, or enable notifications.

Impact

Users who change life circumstances (move country, change name, get married) can't update without falling back to v5. Account hygiene tasks become a bug filing.

Recommendation

Settings is the next page to migrate after AddTransactionModal. Group: Profile · Localization · Appearance · Notifications · Sync · Account · Danger.

02

Admin app — admin/

7 pages, role-gated by Super / Roles / Content. NorthStar dashboard is a strong opening; supporting pages thin out fast. Most "actions" don't act yet.

2.1 — NorthStar dashboard
WatchAD‑01
Dashboard · 12 KPI grid

KPIs are non-clickable · no drill-down · no segment

Each Kpi tile shows value, target, ✓/⚠ — but you can't click "Free → Paid 7.2%" to see which users converted, when, from which surface. Same for D7/D90 retention, multi-member %.

Impact

Admin is a glanceable poster, not an investigation tool. PMs will export to Looker for any real question.

Recommendation

Every KPI links to a filtered Users / Households / Subscriptions list. Add segment toggles at top: Country · Tier · Template · Cohort week. Cohort retention curves > single-number retention.

PolishAD‑02
Dashboard · NPS / MRR / Signups

No anomaly detection · no week-over-week alerts

Signups dropped 20% week-over-week? You'd only know if you happened to look. There's no banner, no "what changed", no comparison to expectation.

Impact

Slow detection of regressions. Stripe webhook fail at midnight = nobody knows till morning standup.

Recommendation

Top-of-page alert strip when any KPI moves > 2σ from 4-week mean. Slack/email push for super admins.

2.2 — Users
CriticalAD‑03
Users · row actions

Row actions are icons with no handlers

Mail, Ban, Shield, MoreHorizontal icons render. None have onClick. Clicking any does nothing. Worse, they're hidden behind opacity-0 group-hover:opacity-100 — but the parent has no group class, so they never appear.

Impact

Support team can see users but can't act on them: no email, no suspend, no role change, no impersonation, no password reset, no refund. Admin is read-only in practice.

Recommendation

Click row → side drawer with Profile · Households · Subscription · Activity · Actions (Email · Suspend · Reset password · Impersonate · Refund · Delete with GDPR confirm). Wire to Supabase admin SDK. Audit-log every action.

WatchAD‑04
Users · table

Pulse Score column has no context for support

Pulse is shown 0–100 with green/yellow/red bands. Support agent fielding "I need help" can't tell why a user is at 41 — is it debt, missed budgets, low savings? They have to switch contexts.

Impact

Support latency. Conversations open with "can you tell me more about your situation" instead of "I see your debt-to-income jumped to 45% last month — want to walk through the planner?"

Recommendation

Hover Pulse → 5-bar breakdown (Budget · Savings · Goal · Trend · Debt). Click → user drawer pre-pinned to Insights tab.

WatchAD‑05
Users · search

Search is name/email/country only

Can't find by Stripe customer ID, Supabase user ID, household name, transaction ID, IP, or signup UTM source — exactly the things support tickets reference.

Impact

Every support ticket starts with "what's your email?"; users who signed up with one address and ticket'd from another can't be found.

Recommendation

Cmd-K global search across users, households, subscriptions, audit, articles. Index on every primary identifier.

2.3 — Households
WatchAD‑06
Households · grid cards

Cards aren't clickable · no drill-in

Card shows type, member count, owner email — but can't open it. No way to see the actual members, who's active, last transaction, total balance, churn risk.

Impact

Multi-member % is the NorthStar lever; admin can't analyse why a household stopped being multi-member.

Recommendation

Click card → drawer with Members table · Activity feed · Subscription · Health metrics. Add filter: "stopped being multi-member in last 30d" (early churn signal).

2.4 — Subscriptions & billing
CriticalAD‑07
Subscriptions · row

No admin actions for billing — refund, comp, retry, cancel

The list shows status (active/past_due/trialing/canceled), monthly amount, renews date. There's no way to: refund, issue credit, comp a month, retry a failed charge, change plan, change payment method, cancel.

Impact

Every billing question becomes a Stripe-dashboard task. Two systems of record. Support agents need Stripe access (security risk) or escalate to engineering (slow).

Recommendation

v8.1 must include: refund (full + partial), retry, comp 1/3/6 months, change plan, cancel-at-period-end, mark fraudulent. All write through Stripe + log to audit.

WatchAD‑08
Subscriptions · top metrics

MRR · ARR · Past Due — but no churn, no LTV, no cohort

The four metric tiles cover today's snapshot. They don't show: monthly churn rate, gross/net revenue retention, LTV by tier, payback period, dunning recovery rate. These are the metrics a CEO opens admin to see.

Impact

Admin can't answer "is the business healthy". CEO will open Stripe + a spreadsheet instead.

Recommendation

Add a Revenue tab: Cohort retention by signup month, MRR movement (new + expansion + contraction + churn = net), Dunning recovery %, LTV/CAC (needs marketing input), 30/60/90-day churn.

2.5 — Content
WatchAD‑09
Content · "New Article" button

No editor · no preview · no scheduling

The list is functional; the create flow doesn't exist. There's no way to write, schedule, or even see a published article — the eye icon's onClick is a no-op. The Content admin role's only reason to exist isn't deliverable.

Impact

Content team can't do their job in the admin tool. They use Notion → eng exports → ships. Multi-week loop for what should be 1-day publishing.

Recommendation

Markdown / Tiptap editor with autosave · live preview · schedule date · topic + tags · author · cover image upload · SEO fields · review state machine (draft → review → published → archived). Page-level analytics inline.

PolishAD‑10
Content · views column

No engagement metrics beyond views

"Views" is the weakest content KPI. No time-on-page, scroll depth, conversions to signup, completion rate, refer-back rate.

Impact

Content team optimises for traffic, not value. Wrong incentive.

Recommendation

Wire PostHog page-view + custom events. Show: signups attributed, time-on-page p50, scroll-depth p50.

2.6 — Audit log
WatchAD‑11
Audit · filters

Substring search only · no filters

For a compliance trail (7-year retention!) the only filter is the global search box. No date range, no actor, no action type, no entity, no IP — exactly the things auditors and forensic investigators ask for.

Impact

SOC 2 / GDPR audit response measured in days, not minutes. Every "show me all role changes by user X in 2025" is a CSV export + spreadsheet.

Recommendation

Filter chips: Date range · Actor · Action · Entity · Role · IP. Saved views ("All role changes", "All deletions"). Export already exists — keep it, but make most queries answerable in-page.

PolishAD‑12
Audit · entity column

Entity rows show ID but don't link

"User u_0142" — clicking should jump to Users filtered to that user. Currently it's text.

Impact

Investigation workflow is copy-paste-search instead of click-through.

Recommendation

Linkify every entity ID to its detail drawer.

2.7 — Settings & security posture
CriticalAD‑13
Settings · Security

No SSO · no 2FA · no IP allowlist · no impersonation audit

"Required for v8.1" is shown as a status pill — meaning launch v8.0 has admins logging in with email/password and full read access to every household's financial data. Combined with no rate limit on the admin API surface this is a P0 security gap.

Impact

Single phished credential = full data breach across all customers. Insurance / SOC 2 / GDPR all blocked. This is also a customer-trust ceiling — "do you do 2FA on internal access?" is question 1 from any enterprise customer.

Recommendation

Block v8.0 launch. Ship Google Workspace SSO + mandatory 2FA + IP allowlist + 30-min idle timeout (already configured) + impersonation audit-log row before any production data lands. Security must be wired before billing actions (AD-07).

WatchAD‑14
Layout · role switcher

Role switcher is in the production sidebar

The "Acting as: Super Admin" dropdown lets you switch tier from the sidebar. This is great for testing. It's a privilege-escalation footgun in production.

Impact

A Content admin who's also a sometime Super gets in the habit of switching down to test their team's view, forgets to switch back, takes a destructive action expecting Content scope.

Recommendation

Behind a VITE_DEV_ROLE_SWITCH env flag. Production = role assigned by SSO claim, immutable for the session.

03

Cross-cutting — consumer ↔ admin · platform

Gaps that aren't in either app individually but in the seam between them, or in things both apps share.

CriticalXC‑01
No support / impersonation bridge

Support can't see what the customer sees

When a user reports "my Pulse is wrong" or "the chart is empty," support has no read-only impersonation. Admin shows aggregated data; customer app shows their data. There's no "view as user" mode.

Impact

Every support ticket needs screenshot + remote share. P75 resolution time inflates 5–10×. With multi-currency + multi-template + multi-household, repro is hard.

Recommendation

"Open as user" button on every Users row → opens consumer app in a tab with read-only banner + impersonation token. Audit-logged. Pair with AD‑13 SSO so it's traceable to the admin.

WatchXC‑02
No notifications system end-to-end

Notification Center exists; no producers

Sidebar imports NotificationCenter. Recurring schedules have a reminderLeadDays field. There's no service worker, no push, no email, no SMS, and no admin view of notification health.

Impact

reminderConfirmedPct > 80% is unmeasurable. The "kitchen-table conversation" depends on out-of-band nudges.

Recommendation

v8.1 wire: web push (PWA), email (Resend/SES), in-app feed. Admin gets a Notifications tab: send rate, open rate, click rate, bounce. Per-user notification settings page in consumer Settings.

WatchXC‑03
Three apps · two design systems

v5 vanilla, v6 React, v8 admin — three codebases, two type systems

Consumer is paper-warm + Newsreader/Inter Tight; admin is Claude-native + Source Serif 4/Inter. The visual jump between them is jarring for users who'll touch both (e.g. an admin with their own consumer account, support staff demoing).

Impact

Brand dilution. Component duplication. Tokens drift over time as different teams maintain them.

Recommendation

Extract shared @finflow/tokens + @finflow/ui packages. Admin keeps Claude-native palette but uses the same Card / Panel / Table primitives. Retire v5 vanilla once stubs are migrated.

WatchXC‑04
Mobile

No native app · web responsive but not optimised for daily use

Consumer is responsive (sidebar collapses, mobile bar). Daily-use behaviour (open phone, scan a receipt, log it in 5 seconds) needs native: camera, biometric unlock, share extension from banks' SMS. Admin doesn't matter on mobile.

Impact

avgSessionsPerUser > 4/wk won't be hit on web alone. Mobile-first competitors will out-stick FinFlow.

Recommendation

PWA first (installable, push, offline cache) before native. Native iOS in v9.

PolishXC‑05
Accessibility

No a11y pass evident

Pulse Gauge is custom SVG with no aria-label / live region. Color is the primary signal across the app (sage/honey/terra). Modal focus trap not visible in code. Date format relies on locale guessing not user preference for assistive tech.

Impact

Multi-gen + elder personas locked out. EU Accessibility Act (June 2025) compliance risk.

Recommendation

Run axe + manual keyboard pass on every page. Add text labels alongside color. Target WCAG 2.2 AA.

PolishXC‑06
Empty & error states

Empty states use emojis · errors silently default

Throughout: <EmptyState icon="◎" />, icon="🔍", icon="🥯". Error states (setHistory, refresh) catch and toast — but no retry button on most network failures.

Impact

Inconsistent voice. Bagel emoji on "no expenses this month" undercuts the Newsreader gravitas the rest of the app earns.

Recommendation

Define 1 illustration set (paper-cut style would match the brand). Standardise: empty = invitation to act ("Add first transaction →"), error = retry CTA + support link.

04

What's already working — don't break

Equally important to call out. Optimise these; don't rebuild.

KeepOK‑01

Pulse Score is the right NorthStar metric

5 components, transparent weights, debt-aware. The single number households can rally around. Just make its inputs editable (CU‑15).

KeepOK‑02

Paper-warm palette is differentiated

Coral + cream actually does feel like kitchen-table not quarterly-statement. It's the strongest brand asset; protect it.

KeepOK‑03

Privacy-first AI summary is a real moat

buildSafeSummary strips merchant names. This is genuine differentiation vs. competitors that ship raw txn rows to OpenAI. Lead with it in marketing.

KeepOK‑04

Role tiers in admin (Super/Roles/Content)

Right number of tiers. Right scoping. The implementation just needs SSO (AD‑13) and the actions to actually do things.

KeepOK‑05

Template-driven onboarding is the right shape

6 templates × pre-populated budgets/goals/debts is the right "blank-page solver." Polish it (CU‑01..03), don't rethink it.

KeepOK‑06

Multi-currency stored at transaction level

12 currencies, per-row currency, base-currency conversion at view time. Correct architecture for global + remittance + multi-gen households.
